# 12 中割り折り（インサイドリバースフォールド）

鶴の首・尾・頭のように、折り重なった1本の「点」の先端を、折り筋を反転させながらレイヤーの間へ差し込む操作（`kind: "insideReverse"`、`applyInsideReverseFoldStep`）。[09](./09-squash-fold.md)・[10](./10-petal-fold.md) と同じく履歴にはジェスチャの最小情報（`foldLine` / `dragVertex` / `viewFront`）だけを持ち、幾何はリプレイ時に再導出する。

## なぜ通常の折りで表現できないか

鶴の基本形＋足細めの先端には、首になる点と尾になる点が**同一の平面位置に重なっている**（表裏あわせて16枚。しかもレイヤー順で交互に混ざっている）。通常の折り（[04](./04-fold-execution.md)）は視点側から数えた枚数で対象を選ぶため、片方の点だけを選べず、破れ判定で「16枚まとめて折る」しか残らない。さらに通常の折りは動く片を常に**視点側の外側**（最大レイヤーの上）へ積むため、首が羽の上に倒れてしまい、中へ差し込む形にならない。

## 動かす点の特定 — 展開図連結成分

dragVertex を持つ候補板を、候補板同士の共有折り目（`findSharedSourceSegments`、[06](./06-tear-prevention.md)）で Union-Find し、**視点側の先頭候補板を含む連結成分**を「動かす点」とする。鶴の先端では首の点（紙の一方の角由来の8枚）と尾の点（反対の角由来の8枚）がちょうど2成分に分かれる。

- 成分が1枚だけの場合は折り重なった点ではないため不成立（通常の折りで表現できる）
- 同一位置に2本の点がある場合、最初の中割りは視点側の先頭板を含む成分が動く（どちらの点を先に起こしても対称）。2本目は残った成分が一意に選ばれる

## 変換と成立条件 — applyInsideReverseFoldStep

1. dragVertex が折り線上にないこと
2. **折り線が点のスパインを横切ること**（`crossesSpineCrease`）。スパイン = 成分内の板同士の共有折り目のうち、折り畳み空間で dragVertex を端点とするもの。中割り折りの本質は折り筋の山谷反転なので、反転する折り筋を折り線が横切らない操作（例: スパインから離れた開いた角を起こす折り）は中割り折りではない。この条件がないと「折り重なった紙の先端をドラッグする」ほぼ全ての操作で成立してしまう（先端片は同一の鏡映を受けるため破れ検証が原理的にほぼ落ちない）
3. 点の各板を折り線で分割し、dragVertex 側を先端片とする（板全体が dragVertex 側にある場合は丸ごと先端片）。少なくとも1枚は分割されること（新しい折り目ができること）
4. **全ての先端片は折り線での鏡映という同一の変換**を受ける（平面上の最終形は通常の折りと同じ。違いはレイヤーだけ）
5. 最終状態で全ての片のペアの共有折り目が折り畳み空間でも一致すること（`isConnectivityPreserved`）。片方の点だけを動かせるのは、2本の点が展開図上で本体を介してしかつながっていないため

## レイヤー — 半分ごとに逆転して間へ差し込む

先端片は外側ではなく、点の前半分と後半分の間へ差し込む:

- 点の板をレイヤー順（視点側から）に並べ、中央で前半分・後半分に分ける
- 前半分の先端片は根元の裏へ、後半分の先端片は根元の手前へ巻き込まれるため、**半分ごとに重なり順を逆転**して積む（例: 点が前から a,b,c,d なら先端片は前から b,a,d,c）
- 挿入位置は前半分の最も奥の板のレイヤー直下の**小数レイヤー**で表す（整数レイヤーの隙間に入れるため。レイヤーは重なり順を表す相対値なので小数でも描画・後続の折りに影響しない）

前後半の分割はレイヤー順の中央による近似で、挿入後の厳密な貫通判定は行わない（エンジン全体の既知の制限 = [01](./01-architecture.md) と同じ）。

## 発動 UX とアニメーション

- [05](./05-multi-board-folding.md) の枚数選択フローに統合。成立する場合は `FoldCountSelector` に「中割り折り」の選択肢が加わる（`FoldChoice = number | "squash" | "petal" | "insideReverse"`）
- 全ての先端片が同一回転を受けるため、アニメーションは通常の折りと同じ**単一ピボット Group 方式**を共用する（`commenceInsideReverseFold` → `replaceSceneForFolding` → `useFoldAnimation` の fold/insideReverse 共通分岐）。差し込みレイヤーは確定後のリプレイ再描画で反映される
- 投稿データ化（[11](./11-procedure-export.md)）も通常の折りと同じ「全頂点が折り線周りの単一回転」として書き出す（`kind: "insideReverse"`）

## 既知の制限

- 動く片は視点側を通って回転する（実物の中割り折りは紙の間を通るが、途中経過の見た目のみの差で最終形は正しい）
- 前後半の分割位置・差し込み順は基本ケース（対称に折り重なった点）を想定した近似
- スパイン条件を満たす場面は多い（折り目の端点の角を折り込む操作は実際の折り紙でも広く可能）ため、選択肢としては比較的よく提示される。差し込み時の貫通判定はエンジン全体と同様に行わない

## 鶴の完成シーケンス

`craneCompletion.test.ts` が固定化している完成手順: 鶴の基本形＋足細め（[craneFixture](../utils/replayFoldSteps/craneFixture.ts)）→ 首（中割り）→ 尾（中割り）→ 頭（首の新しい先端への中割り）→ 羽の折り下げ（表4枚・裏4枚の通常の折り）。計15手・最終72枚。
